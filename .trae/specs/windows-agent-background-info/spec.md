# Windows Agent 后台信息上报 - 产品需求文档

## 概述
- **摘要**：更新 Windows agent，使其能够同时上报前台应用和后台应用信息，并在服务器端和前端进行相应处理，以在网站上显示“正在运行”（后台程序）和“正在查看”（前台程序）的状态。
- **目的**：提供更全面的应用使用情况监控，区分前台和后台应用的状态，提升用户对设备使用情况的了解。
- **目标用户**：使用 Live Dashboard 的用户，特别是需要监控 Windows 设备应用使用情况的用户。

## 目标
- 扩展 Windows agent 功能，使其能够同时上报前台应用和后台应用信息
- 修改服务器端数据接收和处理逻辑，支持后台应用信息
- 更新前端界面，显示“正在运行”和“正在查看”的状态
- 确保数据上报的准确性和实时性

## 非目标（范围外）
- 不修改其他平台（如 macOS、Android）的 agent 实现
- 不改变现有的前台应用监控逻辑
- 不增加额外的系统资源消耗

## 背景与上下文
- 现有 Windows agent 只上报前台应用信息，无法监控后台运行的应用
- 后端已经有 `is_foreground` 字段，但目前默认值为 true
- 前端界面需要根据应用状态显示不同的状态文本

## 功能需求
- **FR-1**：Windows agent 能够同时收集前台应用和后台应用信息
- **FR-2**：Windows agent 能够区分前台和后台应用，并在上报时设置正确的 `is_foreground` 字段
- **FR-3**：服务器端能够正确处理包含后台应用信息的上报
- **FR-4**：前端界面能够根据 `is_foreground` 字段显示不同的状态文本

## 非功能需求
- **NFR-1**：Windows agent 的资源消耗增加不超过 10%
- **NFR-2**：数据上报频率保持不变，不增加服务器负载
- **NFR-3**：前端界面的响应速度不受影响

## 约束
- **技术**：使用现有的 Python 环境和依赖库
- **依赖**：依赖 psutil 库获取进程信息
- **兼容性**：保持与现有配置文件格式的兼容性

## 假设
- 后台应用定义为：当前不在前台运行但正在活动的应用进程
- 只监控用户级别的应用进程，不监控系统进程
- 后台应用的上报频率与前台应用相同

## 验收标准

### AC-1：Windows agent 能够同时上报前台和后台应用信息
- **给定**：Windows agent 正在运行
- **当**：用户打开多个应用，其中一些在前台，一些在后台
- **则**：agent 能够同时上报前台和后台应用的信息，包括进程名和窗口标题
- **验证**：`programmatic` - 检查服务器端是否接收到包含前台和后台应用信息的数据

### AC-2：服务器端能够正确处理后台应用信息
- **给定**：服务器正在运行，Windows agent 上报后台应用信息
- **当**：服务器接收到包含 `is_foreground: false` 的上报数据
- **则**：服务器能够正确存储和处理这些数据，包括在数据库中设置正确的 `is_foreground` 标志
- **验证**：`programmatic` - 检查数据库中的 `ActivityRecord` 和 `DeviceState` 是否正确设置了 `is_foreground` 字段

### AC-3：前端界面能够显示不同的应用状态
- **给定**：前端界面正在运行，服务器端有前台和后台应用的记录
- **当**：用户查看设备状态或时间线
- **则**：前端界面能够根据 `is_foreground` 字段显示“正在查看”（前台）或“正在运行”（后台）
- **验证**：`human-judgment` - 手动检查前端界面是否正确显示应用状态
