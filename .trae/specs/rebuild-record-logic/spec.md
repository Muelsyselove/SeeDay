# Live Dashboard 记录逻辑重构 - 产品需求文档

## Overview

* **Summary**: 重构 Live Dashboard 的记录逻辑，包括客户端提交格式和后端解析逻辑，实现基于前后台状态的应用持续运行判断，以及跨天处理和媒体应用的特殊处理。

* **Purpose**: 解决当前记录逻辑的问题，提高时间记录的准确性和一致性，特别是对于持续运行的应用和跨天处理。

* **Target Users**: 使用 Live Dashboard 的用户，包括需要监控设备活动的个人和团队。

## Goals

* 重构客户端提交格式，使用分号分隔的格式提交多个应用信息

* 重构后端解析逻辑，处理这种新格式

* 实现基于前后台状态的应用持续运行判断

* 实现跨天处理逻辑

* 实现媒体应用的特殊处理逻辑

* 实现详细的显示格式需求

## Non-Goals (Out of Scope)

* 不修改数据库结构

* 不改变 API 接口格式

* 不添加新的功能特性

## Background & Context

* Live Dashboard 使用客户端 agent 收集设备活动数据，包括前台和后台应用

* 当前的记录逻辑基于时间阈值判断应用是否持续运行，不够准确

* 需要改为基于前后台状态的判断逻辑，当应用在一次提交中不在前台或后台运行时视为应用结束

* 需要处理跨天的情况，确保时间记录的准确性

* 需要特殊处理媒体应用，包括音乐和视频应用

* 需要实现详细的显示格式需求，确保界面美观和信息完整

## Functional Requirements

* **FR-1**: 重构客户端提交格式，使用分号分隔的格式提交多个应用信息

* **FR-2**: 重构后端解析逻辑，处理这种新格式

* **FR-3**: 实现基于前后台状态的应用持续运行判断

* **FR-4**: 实现跨天处理逻辑

* **FR-5**: 实现媒体应用的特殊处理逻辑

* **FR-6**: 实现详细的显示格式需求

## Non-Functional Requirements

* **NFR-1**: 数据处理性能应保持不变或有所提升

* **NFR-2**: 重构不应影响现有的其他功能

* **NFR-3**: 重构应保持向后兼容性

* **NFR-4**: 显示格式应美观、清晰、易读

## Constraints

* **Technical**: 基于现有的代码结构和架构

* **Business**: 保持现有功能的完整性

* **Dependencies**: 依赖现有的客户端 agent 数据格式

## Assumptions

* 客户端 agent 能够正确提交新格式的数据

* 现有的 API 接口格式保持不变

* 前端和后端的代码结构可以进行必要的修改

## Acceptance Criteria

### AC-1: 客户端提交格式重构

* **Given**: 客户端 agent 需要提交多个应用信息

* **When**: 客户端 agent 向服务器发送数据

* **Then**: 客户端 agent 应使用分号分隔的格式提交多个应用信息，包括 app\_id、window\_title、is\_foreground 等

* **Verification**: `programmatic`

### AC-2: 后端解析逻辑重构

* **Given**: 服务器接收到客户端 agent 发送的新格式数据

* **When**: 服务器处理这些数据

* **Then**: 服务器应正确解析分号分隔的格式，提取多个应用的信息

* **Verification**: `programmatic`

### AC-3: 应用持续运行判断

* **Given**: 客户端 agent 连续提交数据

* **When**: 服务器处理这些数据

* **Then**: 服务器应基于前后台状态判断应用是否持续运行，当应用在一次提交中不在前台或后台运行时视为应用结束

* **Verification**: `programmatic`

### AC-4: 跨天处理逻辑

* **Given**: 客户端 agent 提交跨天的数据

* **When**: 服务器处理这些数据

* **Then**: 服务器应正确处理跨天的情况，在第一天结束所有条目，在第二天创建新条目

* **Verification**: `programmatic`

### AC-5: 媒体应用特殊处理

* **Given**: 客户端 agent 提交包含媒体应用的数据

* **When**: 服务器处理这些数据

* **Then**: 服务器应特殊处理媒体应用，包括音乐和视频应用，按照规则显示到媒体使用栏并记录时间

* **Verification**: `programmatic`

### AC-6: 显示格式需求

* **Given**: 服务器处理客户端 agent 发送的数据

* **When**: 服务器生成显示数据

* **Then**: 服务器应按照要求的格式显示应用信息，包括运行时间、总运行时间等

* **Verification**: `human-judgment`

## 显示格式需求

### 应用运行状态显示

* **持续运行的应用**：显示为 "now  起始时间（xx:xx）-现在 应用名称"

* **已结束的应用**：显示为 "起始时间（xx:xx）-结束时间（yy:yy） 应用名称"

* **跨天的应用**：

  * 第一天结束的条目：显示为 "起始时间（xx:xx）-次日 应用名称"

  * 第二天新建的条目：显示为 "now  昨日-现在 应用名称"

### 时间记录显示

* **当天条目的运行时间**：在条目末尾记录

* **该应用在今天的总运行时间**：在条目末尾记录

### 媒体应用显示

* **音乐应用**：

  * 显示到媒体使用栏

  * 不同音乐分开记录

  * 同一音乐合并时间

* **视频应用**：

  * 如果在前台运行，显示到媒体使用栏并记录时间

  * 如果不在前台运行，显示到媒体使用栏但不累计时间

  * 不同视频分开记录

  * 同一视频合并时间

## 详细逻辑实现

### 1. 客户端提交格式

* **app\_id**：多个应用的进程名称，用分号分隔（如 "A;;B;;C;;D"）

* **window\_title**：多个应用的窗口标题，用分号分隔，与 app\_id 按顺序对应（如 "A-XXX;;B-XXX;;C-XXX;;D-XXX"）

* **timestamp**：时间戳，格式为年;月;日;xx:xx（如 "2026;04;16;11:27"）

* **is\_foreground**：多个应用的前后台状态，用分号分隔，与 app\_id 按顺序对应（如 "1;0;0;0"，1 代表前台，0 代表后台）

* **extra**：额外信息，如电池状态、音乐信息等

### 2. 后端解析逻辑

* 解析分号分隔的格式，提取多个应用的信息

* 处理 timestamp 格式，提取年、月、日、时间信息

* 对于每个应用，创建或更新对应的记录条目

### 3. 应用持续运行判断

* 第一次提交时，为每个应用创建新的记录条目，记录起始时间

* 后续提交时，检查应用是否在提交的应用列表中

  * 如果在，视为持续运行，更新结束时间

  * 如果不在，视为结束运行，记录结束时间

### 4. 跨天处理逻辑

* 检查提交的 timestamp 中的年、月、日信息

* 如果与上一次提交不在同一天：

  * 在第一天结束所有条目，记录结束时间为第一天的 24:00，但显示结束时间为次日

  * 在第二天为仍在运行的应用创建新的记录条目，记录起始时间为第二天的 00:00，但显示起始时间为昨日

  * 在第一天最后一次提交时仍运行，但第二天第一次提交时没有运行的应用结束条目，记录结束时间为24:00

### 5. 媒体应用特殊处理

* 识别音乐应用和视频应用

* 对于音乐应用：

  * 显示到媒体使用栏

  * 记录运行时间，不同音乐分开记录，同一音乐合并时间

* 对于视频应用：

  * 如果在前台运行，显示到媒体使用栏并记录时间

  * 如果不在前台运行，显示到媒体使用栏但不累计时间

  * 不同视频分开记录，同一视频合并时间

### 6. 时间计算逻辑

* **运行时间**：结束时间 - 起始时间

* **总运行时间**：当天所有该应用条目的运行时间之和

* **跨天时间**：第一天的运行时间为第一天结束时间 - 第一天起始时间，第二天的运行时间为第二天结束时间 - 第二天起始时间
